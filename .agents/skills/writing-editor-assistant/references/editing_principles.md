# 编辑原则与审稿标准

本文档定义了技术博客编辑的核心原则和审稿标准，为 writing-editor-assistant 提供指导。

## 核心编辑原则

### 1. 读者视角原则

**预判困惑**：
- 在读者可能困惑的地方提前解答
- 不要假设读者具备所有前置知识
- 在引入新概念时，提供必要的背景信息

**提供背景**：
- 不假设读者具备所有前置知识
- 在首次出现术语时，提供定义和解释
- 适当引用参考资料，帮助读者深入理解

**循序渐进**：
- 从简单到复杂，从具体到抽象
- 先建立直觉，再深入原理
- 每个章节都应该建立在前面内容的基础上

**示例**：
```markdown
❌ 不好的写法：
"使用 RAII 可以自动管理资源。"

✅ 好的写法：
"RAII（Resource Acquisition Is Initialization）是 C++ 中一种自动管理资源的技巧。其核心思想是：资源的获取和释放与对象的生命周期绑定。当对象创建时获取资源，当对象销毁时自动释放资源。这样我们就不需要手动记住释放资源，避免了资源泄漏。"
```

---

### 2. 逻辑严谨原则

**概念清晰**：
- 每个核心概念都有明确定义
- 避免模糊不清的表述
- 区分事实和观点

**论证完整**：
- 从前提到结论，每一步都有依据
- 不跳跃关键推理步骤
- 必要时提供证明或引用

**无逻辑跳跃**：
- 段落之间有过渡
- 章节之间有衔接
- 从一个话题转向另一个话题时，解释二者的关系

**示例**：
```markdown
❌ 不好的写法：
"Rust 的所有权机制很安全。因此我们应该使用 Rust。"

✅ 好的写法：
"Rust 的所有权机制通过编译时检查，确保了内存安全。它在编译阶段就能发现数据竞争、悬垂指针等问题，而不是等到运行时才崩溃。因此，对于需要高可靠性的系统级编程，Rust 是一个更好的选择。"
```

---

### 3. 技术深度原则

**触及原理**：
- 不仅说"怎么做"，更说"为什么"
- 解释设计背后的权衡和考量
- 指出技术的适用边界

**避免表面化**：
- 不满足于 API 文档式的介绍
- 不停留在"怎么调用这个函数"
- 深入到"为什么要这样设计"

**提供洞察**：
- 指出设计背后的权衡和考量
- 对比不同方案的优劣
- 给出实践中的经验和教训

**示例**：
```markdown
❌ 不好的写法（表面化）：
"使用 `std::move` 可以提升性能。"

✅ 好的写法（有深度）：
"`std::move` 本身并不提升性能，它只是做了一个类型转换（将左值转换为右值引用）。真正的性能提升来自于移动构造函数和移动赋值运算符，它们可以'窃取'源对象的资源，而不是深拷贝。

例如，`std::vector` 的移动构造函数只需要复制几个指针，而拷贝构造函数需要复制所有元素。当你有一个包含 100 万元素的 vector 时，这个差异是巨大的。

但是要注意：如果类型没有定义移动构造函数（比如 `int`），使用 `std::move` 不会有任何性能提升，反而可能让代码更难理解。"
```

---

### 4. 简洁表达原则

**删除冗余**：
- 每句话都有存在的价值
- 删除重复的表述
- 避免过度铺垫

**用词精确**：
- 技术术语使用准确，不模棱两可
- 避免"可能"、"大概"、"好像"
- 不确定时，明确说明是推测

**结构清晰**：
- 标题层级合理
- 段落长度适中
- 使用列表、图表辅助理解

**示例**：
```markdown
❌ 不好的写法（冗余）：
"在本文中，我们将会讨论并且学习关于 C++ 中智能指针的使用方法和相关技巧，特别是 `std::unique_ptr` 这个智能指针的用法。"

✅ 好的写法（简洁）：
"本文讨论 C++ 智能指针的使用，重点是 `std::unique_ptr`。"
```

## 审稿标准

### 致命问题（必须修复）

这些问题会导致读者无法理解文章，或者产生错误的认知：

- **概念定义模糊或错误**：
  - 核心概念没有明确定义
  - 概念定义与业界标准不符
  - 术语使用不一致

- **逻辑跳跃导致无法理解**：
  - 从前提直接跳到结论，缺少中间步骤
  - 段落之间没有过渡，读者不知道为什么要讲这个
  - 引入新概念时没有解释

- **技术细节不准确**：
  - 代码无法运行
  - 代码与描述不符
  - 技术陈述有误（会被读者指出错误）

- **缺少关键信息**：
  - Bug 复现步骤不完整
  - 配置说明缺少关键参数
  - 代码示例缺少必要的上下文

**如何指出致命问题**：
```markdown
### 【致命问题】概念定义模糊

**问题**：第3节提到了"所有权"和"借用"概念，但没有给出明确定义。读者可能会困惑这两个概念到底是什么。

**位置**：第3节，"所有权和借用"段落

**影响**：这是 Rust 的核心概念，如果读者不理解，后面内容都难以理解。

**建议**：
- 在第3节开头加一段："所有权（Ownership）规定每个值都有一个所有者，且同一时间只能有一个所有者。当所有者离开作用域时，值会被自动释放。借用（Borrowing）允许我们引用值而不获取所有权。"
- 举一个简单的例子说明所有权转移
```

---

### 改进建议（建议修复）

这些问题不影响理解，但修复后可以提升文章质量：

- **结构不够清晰**：
  - 章节顺序不合理
  - 标题层级混乱
  - 缺少总结性段落

- **表达可以更精炼**：
  - 句子过长，难以理解
  - 重复表述
  - 过度铺垫

- **缺少例子或类比**：
  - 抽象概念没有具体例子
  - 缺少生活化类比
  - 代码示例不够典型

- **技术深度可以提升**：
  - 只说"怎么做"，没说"为什么"
  - 没有触及设计原理
  - 缺少对比分析

**如何提出改进建议**：
```markdown
### 【改进建议】结构

**问题**：第4节"实际应用"放在了"数学本质"之前，但实际应用依赖于数学本质的理解。

**建议**：交换第3节和第4节的顺序，先讲数学本质（理论），再讲实际应用（实践）。

**为什么**：读者需要先理解标签分类的多维坐标本质，才能理解为什么它适合搜索引擎的精准检索。
```

---

### 可选优化（锦上添花）

这些是可选的，优化后可以让文章更完美：

- **标题可以更吸引人**：
  - 当前标题平淡
  - 可以用问句、对比等方式吸引读者

- **图表可以更美观**：
  - ASCII 艺术图可以换成 PlantUML
  - 表格可以美化

- **参考资料可以更丰富**：
  - 添加官方文档链接
  - 添加相关论文链接
  - 添加扩展阅读

**如何提出可选优化**：
```markdown
### 【可选优化】标题

**当前标题**：标签分类的使用场景

**优化建议**：从图书馆到搜索引擎：为什么现代系统偏爱标签分类？

**为什么**：原标题只是陈述事实，优化后的标题通过对比（图书馆 vs 搜索引擎）和提问（为什么），更能吸引读者点击。
```

## 审稿优先级

当同时发现多个问题时，按以下优先级指出：

1. **优先致命问题**：这些问题会导致读者无法理解或产生错误认知
2. **其次改进建议**：这些问题影响阅读体验
3. **最后可选优化**：这些问题是锦上添花

**示例审稿反馈结构**：
```markdown
## 【致命问题】
（如果有，必须修复）

## 【改进建议】
（建议修复，提升质量）

### 结构与逻辑
### 技术深度
### 表达清晰度
### 读者体验

## 【亮点】
（值得肯定的部分）

## 【可选优化】
（锦上添花）
```

## 审稿心态

**建设性而非批判性**：
- 指出问题的同时，给出具体的改进方向
- 不要只说"不好"，要说"如何改进"
- 理解作者的创作意图，在此基础上提供建议

**技术内容优先**：
- 关注技术和表达，而不是个人风格
- 允许作者有自己的写作风格
- 只要清晰、准确，不同的表达方式都是可以接受的

**尊重作者的决策**：
- 所有建议都是可选的
- 最终决策权在作者
- 如果作者不同意某条建议，可以讨论，但不强制
