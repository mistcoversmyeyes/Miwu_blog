```md
# 1) 通信协议：我们是怎么聊出来“四维框架”的

你：我之前写过草稿，把协议设计分成语法、语义、时序。  
我：这个框架很好，但你这次关于“上下层关系”的洞察很关键，建议加第 4 维：服务契约/跨层关系。  
你：也就是设计协议不只看自己，还要看和上下层怎么配合。  
我：对。向下依赖什么、向上承诺什么，要写清楚。  
你：第 4 点可以展开成差错控制、拥塞控制、流控、有序、完整、时延这些。  
我：完全正确，而且可以按“正确性/安全性/性能性/稳定性/可运维性”分组。  
你：这其实有点哲学意味，系统是联系的。  
我：工程里就是“分层 + 契约 + 权衡”，你这个理解非常成熟。

最后我们达成：
- 协议设计 = 语法 + 语义 + 时序 + 服务契约（跨层）
- 上层通常继承下层服务语义，但要明确“继承了什么、改变了什么、新增了什么”
```

```md
# 2) Nginx：你从“端口映射”一路推到“反向代理中转层”

你：Nginx是不是把本机多个服务都映射到80/443，相当于应用层多路复用？  
我：方向对。更准确说是 Nginx 在做 L7 反向代理。  
你：它怎么知道该转发到哪个服务物理端口？看应用层报文信息吗？  
我：是，主要看 Host/SNI/path，然后命中 server/location，再 proxy_pass 到后端端口。  
你：所以它是不是在本地创建多个虚拟主机文件夹？  
我：这只对静态站点成立。动态站点一般是反代，不是文件夹映射。  
你：那我可以总结成：静态是代取服务，动态是牵线搭桥并监控流量。  
我：这个总结很到位。

最后我们达成：
- 静态站点：Nginx自己读文件并返回
- 动态站点：Nginx转发到后端 + 做流量治理（日志、限流、超时、转发头等）
- 你的 OpenClaw 场景属于第二种
```

```md
# 3) HTTPS：你把握手过程从“细节困惑”梳理成了稳定叙述

你：HTTPS是不是先建单向，再建反向？Finished是不是三次握手？  
我：不是。TCP三次握手早已完成全双工，TLS Finished是完整性校验，不是再建连接。  
你：懂了，TLS是在TCP这个可靠全双工信道之上再包一层安全信道。  
我：对，这句话非常关键。  
你：那密钥协商是不是本质不变，只是报文聚合减少往返？  
我：对，TLS1.3主要是流程压缩，原理还是临时公钥交换 + 共享秘密 + 派生密钥。  
你：HKTF 是什么？  
我：是 HKDF，用来把共享秘密派生成会话密钥。  
你：那面试里我不用背数学，只要说双方都能独立算出同一共享秘密再派生会话密钥。  
我：完全可以，这个表述很专业了。

最后我们达成：
- HTTPS = HTTP over TLS over TCP/IP
- TCP负责可靠全双工；TLS负责认证/机密性/完整性
- TLS1.3主线：ClientHello -> ServerHello/证书 -> ECDHE -> HKDF -> Finished -> 加密传输
```
