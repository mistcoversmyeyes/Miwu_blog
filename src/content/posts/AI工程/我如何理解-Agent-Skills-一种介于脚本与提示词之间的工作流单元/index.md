1. 文章结构建议

```markdown
## 引子：为什么我想重新定义 Agent Skills
- 先交代语境：这不是从官方定义出发，而是从反复使用后的困惑出发
- 点明本文目标：把两点感性认识整理成更稳定的理性判断
- 给出读者预期：读完后，读者至少能回答“Skill 到底是什么”以及“它和脚本、提示词是什么关系”

## Agent Skills 到底是什么
- 先澄清一个常见误解：Skill 不只是提示词文件
- 给出你的定义：可复用的工作流提示词 + 工作流中调用的工具 + 静态资源
- 分别解释三者的职责
- 顺势引出一个很有辨识度的判断：subagent 可以视作处理不确定性的工具，scripts 可以视作处理确定性的工具

## 它为什么既不像传统脚本，也不像纯工作流提示词
- 传统脚本的优点：稳定、可验证、边界清晰
- 纯文本工作流提示词的优点：灵活、开放、能处理模糊任务
- 两者各自的问题：前者太硬，后者太飘
- 引出你的核心判断：Agent Skills 处在两者之间，负责动态平衡确定性与不确定性

## 从“夹在中间”到“平衡两端”
- 解释什么叫确定性：输入、步骤、输出都相对可控
- 解释什么叫不确定性：问题模糊、路径开放、需要模型判断
- 说明为什么 Skill 不是简单拼接 prompt 和 tool，而是在工作流层面重新分工
- 可以举 1 个你亲身遇到的例子：什么时候纯 prompt 不稳，什么时候纯 script 又显得过度僵硬

## 这套理解会怎样影响我设计 Skill
- 不再把 Skill 看成“一个更长的 prompt”
- 而是看成“一个能组织模型能力、工具能力、静态知识的工作流封装”
- 进一步讨论：哪些步骤应该固化成脚本，哪些步骤应该交给 agent/subagent，哪些知识应该沉淀为资源文件

## 结语
- 回到“轮椅”这个比喻，但要落回工程语境
- 强调 Skill 的价值不是替代思考，而是重构人与模型的协作边界
- 可以留下下一篇文章的钩子：如何设计一个真正好用的 Agent Skill
```

2. 避免的陷阱

❌ 不要把全文写成定义罗列，这篇文章最有价值的部分是“这些认识是怎么从实践里长出来的”
❌ 不要只停留在“轮椅”这个比喻上，必须把比喻落回确定性/不确定性、prompt/tool/resource 的分工
❌ 不要把 subagent 和 scripts 写成简单对立，它们更像是应对不同类型问题的两类工具
❌ 不要写成“我已经完全想明白了”的口气，保持“我在形成方法论”的学生视角会更有说服力

3. 建议的写作风格

✅ 从一次真实使用体验切入：比如你在哪个时刻意识到“Skill 不是 prompt 的附件”
✅ 用“定义 - 边界 - 例子”的顺序推进，让抽象判断始终落地
✅ 多写“为什么这样划分”，少写“它还能做很多事”，避免文章发散
✅ 允许保留一点未完成感，这会让文章更像一次认知升级，而不是教科书式定论
