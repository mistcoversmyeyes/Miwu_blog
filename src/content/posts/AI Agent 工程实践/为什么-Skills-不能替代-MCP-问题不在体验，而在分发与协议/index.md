为什么 skills 无法替代 mcp 这个问题，我的观点：skills 虽然能够通过一些手段获取和 mcp 相似甚至一致的使用体验，但是 skills 由于获得这种体验需要依赖于确定性的脚本或者 cli 工具，而这些脚本的依赖于特定平台，平台兼容性差，不容易分发，所以不能够代替仅依赖 https/sse stdio 和一个支持以上协议的 mcp 服务器和客户端 的mcp 生态。一句话说就是即使 skills 能够实现 mcp 的体验，skills 分发的体验由于依赖过多，依然差于mcp。mcp 作为一个依赖少，契约严谨的协议，具有不可替代的地位。

## 引子：为什么我会产生“Skills 能不能替代 MCP”这个问题

- 先交代语境：在持续使用 Skills 的过程中，我确实感受到它能通过 prompt、脚本、CLI 和静态资源的组合，做出和 MCP 很像的使用体验。
- 这种“像”不是幻觉。无论是复用工作流、封装工具调用，还是降低重复操作，Skills 都能在很多场景下逼近 MCP。
- 但问题也正是从这里开始：**体验相似，是否就意味着可以替代？**
- 我一开始也倾向于把这个问题想得太简单，后来才发现，真正决定“能否替代”的，不是表面交互，而是背后的依赖模型、分发成本和协议契约。
- 所以这篇文章想讨论的不是“Skills 能不能模仿 MCP”，而是：**为什么即使它能模仿出类似体验，它依然不能替代 MCP。**

## Skills 为什么会看起来像 MCP

- 先承认一个事实：Skills 和 MCP 在“使用感受”上确实存在明显重叠。
- 对使用者来说，二者都可以把一串原本零散的能力组织起来，让 Agent 在更高层级上完成任务。
- Skills 可以把提示词、脚本、CLI 工具和静态资源打包成一个可复用工作流；MCP 则可以把外部能力通过协议暴露给客户端调用。
- 如果只看最终效果，二者都像是在给模型提供“额外能力”。
- 但这种相似性更多停留在**体验层**，而不是**抽象层级**。体验接近，并不自动推出二者在工程位置上等价。

## Skills 的边界：它依赖的是本地工作流，不是统一协议

- Skills 的基础非常明确：prompt、脚本、CLI 工具、静态资源。
- 这套组合的优点是灵活，能快速把本地已有能力拼装起来；但它的问题也同样明显：它天然依赖宿主环境。
- 一旦一个 Skill 想要复现 MCP 的某种能力，它往往需要借助确定性的脚本或者本地 CLI 工具来补足执行层。
- 而这些脚本和工具通常都依赖具体平台，比如操作系统差异、shell 环境差异、路径结构差异、依赖安装方式差异。
- 结果就是：**Skills 越想获得接近 MCP 的体验，越可能加重自己对本地环境的依赖。**
- 这种依赖并不意味着 Skills 不好，而是说明它更适合做“本地工作流封装”，而不是承担统一能力接入的职责。

## MCP 为什么不可替代：它提供的是契约，而不是某种“技巧”

- MCP 真正关键的地方，不是“也能调工具”，而是它提供了一层明确、稳定、可协商的协议契约。
- 只要双方遵守同一协议，客户端和服务器就可以围绕 stdio、HTTP、SSE 这些通道完成能力接入与调用。
- 这意味着 MCP 的核心前提更少：你不需要为每个场景重新约定脚本怎么写、命令怎么装、环境怎么配，只需要有一个兼容的 MCP Server 和一个兼容的 MCP Client。
- 这种设计带来的价值，不只是“好用”，而是更强的可移植性、可复用性和可分发性。
- 换句话说，MCP 提供的不是一种工作流技巧，而是一种让能力可以被标准化接入的**契约层**。

## 真正的分界线：体验可以接近，但生态基础不同

- 到这里，问题其实已经清楚了：Skills 和 MCP 可以在体验层接近，但它们站在不同的工程层次上。
- Skills 关注的是“如何把当前环境里的能力尽量组织起来，让事情做成”。
- MCP 关注的是“如何让能力通过统一协议暴露出来，并被不同客户端稳定接入”。
- 前者更贴近本地工作流，后者更贴近协议生态。
- 所以判断 Skills 能不能替代 MCP，不能只看“能不能做出差不多的效果”，而要看“能不能以类似的依赖成本、分发成本和契约强度提供同等级别的能力接入”。
- 从这个角度看，答案是否定的：**Skills 可以逼近 MCP 的体验，但无法逼近 MCP 的协议生态基础。**

## 结论：Skills 不能替代 MCP，但二者可以共存

- 如果把问题说得更准确一点，那么结论不是“Skills 不如 MCP”，而是“Skills 和 MCP 解决的问题不是同一层的”。
- Skills 更像是一种工作流封装机制，擅长把 prompt、本地脚本和工具拼起来，让 Agent 在具体环境中更高效地做事。
- MCP 更像是一种能力接入协议，擅长用更少的环境假设和更清晰的契约，把外部能力稳定地暴露给客户端。
- 因此，Skills 当然可以在很多场景中获得和 MCP 相似的体验，甚至可以在局部场景下替代掉某些“看起来像 MCP 的东西”。
- 但这不等于它能够替代 MCP 本身，因为它替代不了协议、分发和跨平台兼容性所构成的那套生态基础。
- 更稳妥的理解应该是：**Skills 可以消费 MCP，也可以补足 MCP 没覆盖的本地工作流，但它不是 MCP 的替代品。**

## 写作时需要避免的陷阱

- 不要把这篇写成“Skills 比 MCP 差”的优劣比较。真正的问题是二者抽象层级不同。
- 不要把“依赖脚本和 CLI”写成纯缺点。那是 Skills 的边界，不是它的原罪。
- 不要只说分发麻烦，却不解释为什么 MCP 的分发更稳。重点在于协议统一、环境假设更少、客户端和服务端解耦。
- 不要使用“更先进”“更优雅”这种空话。要落到少依赖、强契约、跨平台、可复用这些工程事实上。

## 可继续展开的几个问题

- 如果只在单一平台、单一团队内部使用，Skills 的分发劣势是否会被显著削弱？
- “协议” 与 “工作流封装” 的边界应该如何划分？
- Skills 最适合补哪一类 MCP 不擅长处理的本地任务？
- 如果未来某些 Skills 生态引入更强的标准化约束，它会不会向 MCP 靠近？
